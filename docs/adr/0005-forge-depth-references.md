# Deepen forge with internal reference modules

ADR 0001 chose self-containment over depth — `forge` would not depend on superpowers or Matt Pocock skills, even though those skills had accumulated years of refinement. That decision still stands: the user experience of "install one plugin and it works" is part of mekaknight's identity, and re-introducing 11 runtime dependencies undoes that. But the cost of self-containment surfaced quickly — when the user asked whether `forge` covers deep daily development (not only production gating), the honest answer was "the phases are shallower than the embedded skills they replace." That gap is the problem this ADR addresses. We decided to **absorb depth into the plugin itself** through a `skills/forge/references/` subfolder: forge's `SKILL.md` remains the orchestrator, and each phase's underlying discipline lives in a dedicated reference document the orchestrator points to. The first iteration ships two modules — `tdd-discipline.md` and `verification.md` — covering the two phases that daily development hits most often. Subsequent iterations will add modules for grilling, diagnosis, prototyping, planning, architecture, peer review, subagent dispatch, and finishing.

## Considered Options

1. **Inline the depth in `SKILL.md`** — Rejected because: forge's `SKILL.md` is already ~380 lines and serves as the orchestrator; folding 9 discipline modules' worth of detail inline would bloat it past the point where it can be skimmed end-to-end, and would mix orchestration concerns (which phase comes next, what the exit gate is) with discipline concerns (how to actually do TDD well). The orchestrator-vs-discipline distinction is exactly why the references folder exists.

2. **Optional hybrid — detect external skills at runtime and use them if present, fall back to internal patterns otherwise** — Rejected for the same reasons ADR 0001 rejected it: two code paths means two behaviors to maintain and test; users get inconsistent experiences depending on their plugin setup; debugging "why did forge act differently for me" becomes a nightmare. The hybrid pitch is seductive ("you get both!") but the cost falls on the maintainer and on every user who hits the inconsistency.

3. **Modular `references/` subfolder, pointed to from `SKILL.md`** (chosen) — `SKILL.md` is the orchestrator; `references/<topic>.md` is the discipline library. The orchestrator gets a one-line pointer at each phase: "for the deeper discipline, see references/X.md." Agents and humans following forge can read the pointer when they need depth and skip it when they do not. The plugin owns every line, so no version drift; the directory structure makes future iterations additive (drop a new module, add a pointer line, done).

## Consequences

- **First iteration scope is two modules.** `references/tdd-discipline.md` deepens the Build phase; `references/verification.md` deepens the cross-cutting verification gate. The plan file at `docs/plans/jameskyeong/forge-depth-references-v1.md` records the exact scope and the explicit non-goals.
- **Ownership cost is real.** Every reference module is content the mekaknight repo now maintains. Upstream improvements in superpowers or Matt Pocock skills will not flow in automatically — we pull patterns we like through manual review and rewrite. This is the tradeoff ADR 0001 already accepted; the references just make the surface where the cost lives more explicit.
- **The orchestrator stays slim.** `SKILL.md` does not duplicate reference content. The pointer lines are intentionally short — one sentence per reference, naming what the document covers. This preserves `SKILL.md`'s readability as an orchestration spec.
- **Future iterations are additive, not restructuring.** Adding `references/grilling.md` does not change forge's phase structure or its existing references; it adds a new pointer at the Clarify phase. The same pattern applies to diagnosis, prototyping, architecture, planning, subagent patterns, peer review, and finishing.
- **DAILY route is a separate decision.** The user originally asked for forge to serve daily development, not only production gating. Deepening the references is one half of the answer (the discipline is now thorough enough to support daily work); a lighter-weight route that skips Plan / Peer-review / Verify for small daily changes may be the other half. That decision is deferred to a separate ADR after the references iteration matures.
- **The "shallow" criticism in ADR 0001 is partially retired.** ADR 0001 explicitly acknowledged: "forge's clarify/build/review/verify phases will initially be shallower than grill-me/tdd/verification-before-completion." With this ADR, that acknowledgment becomes phase-specific: Build and verification are no longer shallow (they have full discipline references); the other phases remain shallow until their references land in subsequent iterations.
- **No change to user-facing surface.** `/forge` invocation, route structure, phase order, and exit gates are unchanged. The references add depth at the discipline layer without altering the orchestration contract.

## Status

Active. First iteration (TDD + verification) ships with this ADR. Subsequent iterations are tracked as separate plans under `docs/plans/jameskyeong/forge-depth-references-vN.md`.
